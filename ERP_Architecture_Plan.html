1. System Overview

This document defines the complete architecture for mcaffeine's internal ERP platform — covering production planning, supply chain, procurement, manufacturing, QC, finance, vendor & manufacturer portals, and reporting. The system is designed as a domain-driven microservices platform running on Kubernetes, with a Kong API Gateway, Redis cluster, and PostgreSQL per service.

Design goal: Support 1,000 concurrent internal users at launch with a clear, low-friction path to 50,000+ users without rewriting any service — only adding infrastructure.

3. Microservices — All 15 Services

Each service owns its domain, has its own database, and communicates with others via the API Gateway (sync) or BullMQ on Redis (async events and job queues). No service directly queries another service's database.

1. Auth Service
Authentication & Authorization
  • JWT tokens: 15-min access, 7-day refresh (stored in Redis)
  • RBAC roles: admin, planner, procurement_manager, finance, qc_lead, vendor, manufacturer
  • Token blacklist in Redis on logout
  • SSO-ready (SAML / OAuth2 extensible)
DB: auth_db  |  Redis: session store, blacklist
2. Master Data Service
Single Source of Truth
  • SKUs, BOMs (Bill of Materials), UOMs
  • Raw materials (RM) and packaging materials (PM)
  • Vendors, manufacturers, warehouses, cost centers
  • All other services call this for reference lookups
DB: master_db — tables: products, bom_headers, bom_lines, vendors, manufacturers, warehouses
3. Production Planning Service
Plans, MRP, Capacity
  • Create and approve production plans linked to SKUs + BOMs
  • MRP engine: explode BOM → auto-generate procurement demands
  • Capacity checks against manufacturing lines
  • Status flow: Draft → Approved → In Progress → Completed
DB: production_db — tables: production_plans, plan_lines, mrp_runs, capacity_slots
4. Procurement Service
FG PO, RM/PM PO, Reconciliation
  • FG PO Creation: Purchase orders to contract manufacturers for finished goods
  • RM/PM Procurement: POs for raw materials and packaging materials
  • Open PO Reconciliation: Match dispatched vs received qty, flag variances
  • PO lifecycle: Draft → Sent → Acknowledged → Partial GRN → Closed
DB: procurement_db — tables: purchase_orders, po_lines, vendor_acks, po_reconciliation
5. Inventory Service
Stock, GRN, Transfers, Reconciliation
  • Inwarding of FG: Goods Receipt Note when finished goods arrive at warehouse
  • RM/PM Stock Reconciliation: Match physical vs system stock at MFG site
  • Internal Transfer: Move stock between warehouses with approval
  • Double-entry inventory ledger (every movement has debit + credit location)
DB: inventory_db — tables: stock_ledger, grn_headers, grn_lines, internal_transfers, reconciliation_reports
6. Manufacturing Service
Tech Transfer, Revamp, New Launches
  • Tech Transfer: Formulation transfer from R&D to manufacturer — versioned, approval-gated, document-driven
  • Product Revamp: Track formulation changes, BOM updates, cost impact assessment
  • New Launches: Project tracker from ideation → sampling → approval → launch
DB: manufacturing_db — tables: manufacturing_orders, tech_transfers, product_revamps, launch_projects
7. QC Service
Quality Control & Compliance
  • Every GRN inward triggers a QC inspection task automatically
  • QC for RM, PM, Finished Goods, and in-process
  • Upload and manage Certificates of Analysis (COAs)
  • Hold/release workflow with approval trail
DB: qc_db — tables: qc_inspections, qc_parameters, coa_records, hold_orders
8. Finance Service
Costs, COGS, Invoices
  • Cost Creation & Approval: Build cost sheets per SKU (RM + PM + conversion + overhead)
  • Real-time COGS: Computed from production actuals; streamed via Redis pub/sub
  • Invoice Management: AP/AR invoices; 3-way match (PO + GRN + Invoice)
  • Finance overview dashboard with live numbers
DB: finance_db — tables: cost_sheets, cost_approvals, cogs_snapshots, ap_invoices, ar_invoices, invoice_matches
9. Logistics Service
Freight & Shipment
  • Freight Approval: Raise freight request → approval by logistics head → booking confirmation
  • Track shipment status and freight costs per PO/GRN
  • Carrier management and rate comparison
DB: logistics_db — tables: freight_requests, freight_approvals, shipments
10. Workflow & Approval Service
Generic Approval Engine
  • Configurable approval chains (sequential or parallel approvers)
  • Used by: POs, cost sheets, internal transfers, freight, QC holds, tech transfers
  • Notification triggers on every state change
  • Immutable audit trail for every approval action
DB: workflow_db — tables: approval_workflows, approval_steps, approval_instances, approval_actions
11. Analytics & Reporting Service
Dashboards, KPIs, Exports
  • Pre-built reports: production efficiency, procurement spend, inventory turns, COGS by SKU, QC rejection rate
  • Custom report builder with drag-and-drop filters
  • Scheduled PDF / Excel exports (Redis job queue)
  • Reads from Elasticsearch (indexed by events from all services)
DB: analytics_db + Elasticsearch cluster
12. Vendor Portal
External — Vendor-Facing
  • Vendors see only their own POs, invoices, and delivery schedules
  • Submit invoices, acknowledge POs, update shipment status
  • Completely isolated from internal APIs — separate auth scope
  • Hosted on separate subdomain: vendors.erp.company.com
Calls internal Procurement + Logistics APIs; vendor-scoped JWT
13. Manufacturer Portal
External — Manufacturer-Facing
  • Manufacturers view production plans assigned to them
  • Access tech transfer documents and QC standards
  • Submit dispatch details and upload COAs
  • Hosted on: mfg.erp.company.com
Calls internal Manufacturing + QC APIs; manufacturer-scoped JWT
14. Notification Service
Email, WhatsApp, In-App
  • Triggered by Redis pub/sub events from all other services
  • Channels: Email (SES), WhatsApp Business API, in-app alerts
  • Templated notifications: PO approved, GRN created, QC hold, freight approved
  • Delivery log with retry on failure
DB: notifications_db — tables: notification_log, templates
15. Document Service
Files, MOM, Action Items
  • Minutes of Meeting (MOM): Create meeting records, assign action items, track resolution
  • Versioned file storage backed by S3/MinIO
  • Link documents to any entity (PO, tech transfer, QC inspection)
DB: documents_db — tables: documents, mom_records, action_items

5. BullMQ — Async Job Queue Architecture

BullMQ replaces Kafka entirely. It is a TypeScript-native, Redis-backed distributed job queue — no separate message broker infrastructure needed. Every microservice can produce jobs; dedicated worker processes consume them independently.

Why BullMQ over Kafka? Kafka requires its own cluster (ZooKeeper or KRaft), ops overhead, and separate client libraries. BullMQ runs on the Redis you already have, has first-class TypeScript support, a built-in dashboard (Bull Board), and handles everything your ERP needs: delayed jobs, retries with backoff, priorities, cron schedules, and concurrency control — all in one npm package.

// Example: Inventory Service produces a job when GRN is created
import { InjectQueue } from '@nestjs/bullmq';
import { Queue } from 'bullmq';

@Injectable()
export class GrnService {
  constructor(@InjectQueue('inventory:grn-events') private grnQueue: Queue) {}

  async createGrn(dto: CreateGrnDto) {
    const grn = await this.grnRepository.save(dto);
    // Notify QC and Finance asynchronously — fire and forget
    await this.grnQueue.add('grn-created', { grnId: grn.id, skuId: grn.skuId });
    return grn;
  }
}

// QC Service worker consumes the event
@Processor('inventory:grn-events')
export class GrnEventsConsumer {
  @Process('grn-created')
  async handleGrnCreated(job: Job<{ grnId: string; skuId: string }>) {
    await this.qcService.createInspectionTask(job.data.grnId);
  }
}

7. End-to-End Data Flow — FG PO → Inwarding → COGS

Illustrates how modules connect in a real business transaction:

  1. Planner creates a Production Plan linked to a SKU and BOM production-service
  2. MRP engine explodes the BOM → auto-generates Procurement Demands as Redis Stream events production-service → redis
  3. Procurement team creates a FG Purchase Order against the demand procurement-service
  4. Approval engine routes the PO through the configured approval chain workflow-service
  5. Approved PO appears in the Manufacturer Portal for acknowledgment mfg-portal-service
  6. Manufacturer acknowledges, raises dispatch; Freight Request created logistics-service
  7. Freight approved by Logistics Head → carrier booked logistics-service + workflow-service
  8. FG arrives at warehouse → GRN created (Inwarding) inventory-service
  9. GRN event triggers QC inspection task automatically qc-service via redis event
  10. QC releases the batch → stock posted to inventory ledger inventory-service
  11. Finance performs 3-way match: PO + GRN + Manufacturer Invoice finance-service
  12. COGS snapshot updated and pushed live to dashboard via Redis pub/sub finance-service → analytics dashboard
